# 技术决策记录

## ADR-001：零框架原生技术栈

- 日期：2026-04-02
- 状态：已采纳
- 决策：使用原生 PHP + PDO + 原生 HTML/CSS/JS 构建系统
- 原因：适配廉价虚拟主机，降低部署与迁移成本，避免框架引入的额外复杂度

## ADR-002：渐进式 MVC 拆分

- 日期：2026-04-02
- 状态：已采纳
- 决策：后台拆分为 `admin/controllers/`、`admin/views/`、`admin/includes/`
- 原因：先在最小改动下提升结构清晰度，避免一次性重构带来的高风险

## ADR-003：统一事件委托模式

- 日期：2026-04-02
- 状态：已采纳
- 决策：前台交互统一使用 `data-*` 属性和集中事件委托
- 原因：降低模板中的内联脚本数量，减少 DOM 与 JS 的耦合

## ADR-004：双模邮件发送

- 日期：2026-04-02
- 状态：已采纳
- 决策：优先使用 SMTP socket 发送，失败时回退到 PHP `mail()`
- 原因：兼顾生产环境可用性与低门槛主机兼容性

## ADR-005：安装向导链路

- 日期：2026-04-02
- 状态：已采纳
- 决策：采用 `install.php -> schema.sql -> installed.lock` 的安装链路
- 原因：统一首次部署流程，减少手工建表和配置出错概率

## ADR-006：统一日志

- 日期：2026-04-02
- 状态：已采纳
- 决策：通过 `shop_log()` 写入 `logs/app.log`，失败时回退 `error_log()`
- 原因：统一日志入口，方便部署后接入采集和监控

## ADR-007：安装向导整合管理员创建

- 日期：2026-04-03
- 状态：已采纳
- 决策：将管理员账号创建整合进 `install.php` 安装向导，消除 `admin/setup.php` 独立入口
- 原因：减少攻击面，统一安装流程，一步到位完成“建表 + 建管理员”
