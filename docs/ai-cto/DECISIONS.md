# 架构决策记录

## ADR-001：零框架原生实现

- 日期：2026-04-02
- 状态：已采纳
- 决策：坚持使用原生 PHP + PDO + 原生前端，不引入重量级框架
- 原因：目标环境是廉价虚拟主机，部署和维护成本必须足够低

## ADR-002：渐进式 MVC 拆分

- 日期：2026-04-02
- 状态：已采纳
- 决策：后台拆分为 `admin/controllers/`、`admin/views/`、`admin/includes/`
- 原因：优先解决单文件后台不可维护的问题，同时避免一次性重写

## ADR-003：统一事件委托模式

- 日期：2026-04-02
- 状态：已采纳
- 决策：前台交互统一使用 `data-*` 属性 + 单一事件委托监听
- 原因：减少模板内联 JS，降低注入风险，提升可维护性

## ADR-004：双模邮件发送

- 日期：2026-04-02
- 状态：已采纳
- 决策：邮件发送优先使用 SMTP socket，未配置时回退到 PHP `mail()`
- 原因：兼顾生产可用性与低门槛部署环境

## ADR-005：安装向导链路

- 日期：2026-04-02
- 状态：已采纳
- 决策：采用 `install.php -> schema.sql -> installed.lock` 的安装流程
- 原因：让项目具备真正的“开箱可部署”能力，减少人工初始化步骤

## ADR-006：统一日志

- 日期：2026-04-02
- 状态：已采纳
- 决策：通过 `shop_log()` 写入 `logs/app.log`，失败时回退 `error_log()`
- 原因：形成统一日志格式，为部署后监控与排障打基础
